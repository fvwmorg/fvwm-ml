Dominik Vogt <dominik.vogt_at_gmx.de> wrote:
>
> I searched up and down the code but I can't find any way how
> simply moving the pointer over the focused window could trigger
> raising or lowering windows. You said that turning of the
> RaiseOverNativeWindows option 'fixes' the problem. This proves
> that the RaiseOrLowerWindow() function is called when the pointer
> enters the window because it is the only place where the option is
> evaluated.

I wonder whether I could put a debugging message in RaiseOrLowerWindow(), or
somewhere else, to try to verify what is doing this. Would this help? Is
this the best place? (Sorry, it's a long time since I looked at the fvwm

> But fvwm does not call this function. I can think of
> several possible explanations:
>
> 1) Exceed raises the window. Is there perhaps some setting in
> Exceed for this?

Not as far as I can tell. Also if I go back to fvwm 2.3.6 (with identical
Exceed settings), I don't have this problem.

> 2) FvwmEvent is doing this.
> 3) FvwmAuto is running and configured to do this.

I don't use either FvwmEvent or FvwmAuto. In fact I can disable every fvwm
module (ps shows only fvwm2 running) and still see this problem.

> 4) The application raises itself. This would mean that this
> occurs only for some applications.

I tried various applications, both local (eg. xterm, rclock) and remote (eg.
emacs running from Solaris).

> 5) Other reasons?
