On 21 Mar 2002 01:07:44 +0100, Dominik Vogt wrote:
>
> The resize arguments add the border dimensions to the final
> geometry. It's very easy to fix, but that probably breaks some
> old setups relying on it(?). Should we make that change? Any if
> yes, should be fix it in the 2.4 branch too?

I think Resize works correctly. I don't think it should be changed.

Currently it resizes the client window, this is usually what is important.
Also, the size of the client is what is shown in the mini-info window.
Both lines are currently equivalent for many applications:

  Resize 300p 300p
  Resize 300c 300c

The only case when the size of the frame not the client window should be
considered is when trying to fit the frame into a rectangle (page), in
this case the size of the client window is not important by intention.
It probably may be done using a new prefix, like 300pf 300pf.

It would be good if Maximize and Resize were consistent from the start on
'p' syntax, but their main intention is different, the first mainly works
on the whole frame, the secont on the client window. Probably Maximize may
support 'c' prefix.

Anyway, there should be a way to resize a window regardless of the
decoration size.
