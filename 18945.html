E> Also, the AutoRaise problem hasn't been fixed in regards to menus: if
E> you pop-down a menu, then the AutoRaise delay expires, the window is
E> raised and obscures the menu. Isn't there any way to fix this?

not reported 'till now.
On a quick look (it's 0:26) my FvwmAuto doesn't do that.

            +-------------+
       3 | 2 |
            | |
            | |
            | |
   +-------------+ |
   | 1 | |
   | |--------+
   | |
   | |

move from 1 over 2 to 3, open menu, before 2 is raised,
(menu overlaps both windows just a little bit, so you can tell the exact
order)
stay in menu as long as you like, nothing is raised over the menu.
exit menu -> 2 is raised over 1.

The raise is delayed until the menu is closed, no strange behaviour found.


E> Finally, something that needs to be fixed in Fvwm.tmpl: in order for
E> the Solaris dynamic linker to find the X libraries, I needed to change
E> the FvwmModuleTarget and FvwmProgramTarget from '$(CC) -o $_at_...' to
E> '$(CCLINK) -o $_at_...'. This is very important! It makes sure the
E> LD_LIBRARY_PATH gets set before the link so the dynamic linker can find
E> the X libraries. Prior to making this change, 'fvwm2' would not run,
E> with the linker complaining about not be able to find the X libraries,
E> and 'ldd' confirmed it. I tracked it down to this oversight in 'Fvwm.tmpl'.
E> Can this be fixed?

I still have to rename all FvwmManPath settings to FvwmManDir settings,
but X11R4 is the least common denominator here (mix of X11R4, R5 & R6 sun
workstations).
So this could be our local setup.
