>>>>> "Shane" == Shane Holder <holder_at_convex.com> writes:

Shane> I'm using fvwm2 on a HP/712 running 9.05, and every once in a
Shane> while my configuration will come up, and things won't be quite
Shane> right. I won't have buttons on some windows. I may have to
Shane> physically place windows eventhough they should be auto placed.
Shane> I sometimes even loose mouse focus all together, and am forced
Shane> to cut and paste letters into my xterm, which I use as my magic
Shane> cookie, so that I can log out.

Shane> It's as though fvwm was multi-threaded and one thread is
Shane> reading the configuration file, and another thread is starting
Shane> up the system, and there's no synchronization between the two,
Shane> ie. the reading thread doesn't finish before the configuration
Shane> thread starts.

Shane> I am using the FvwmM4 stuff to pre-process my .fvwm2rc file, is
Shane> it possible that fvwm isn't waiting for this processing to
Shane> finish before continuing on?

Shane> Am I the only one to see this behavior?

Apparently. But it might be related to the occasional odd XEmacs
display behavior problems that are being reported recently. Perhaps
there is something happening that is causing a race condition in the X
server or something like that.

Would everyone that is seeing odd problems like this please run fvwm2
with the -debug parameter and see if the problems continue. All this
does is run XSynchronize(dpy,1);, which may cause problems to more
clearly show up, or go away, or might not show anything. I'm merely
curious.

Another possibility - what level of optimization is everyone compiling
with? -O3 and greater sometimes causes strange problems with
programs, so I'd say keep it at -O2 or just -O.

Thanks,
Chuck
