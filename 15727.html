On Wed, 6 Mar 1996, Romano Giannetti wrote:

[...]
> The "more" is that till in this version there is some problem with timed
> events.
> AddToFunc "Move-or-Raise" "I" Raise
> + "M" Move
> + "D" RaiseLower
> I think this is a known problem, but... for example if you have the
> standard "raise or move" bound to the title bar, and you click,
> wait the right bit, :-) , and hop, the windows got moved even if you
> have released the mouse button and follow you.

I've encountered this as well. I used to have ClickTime set to 50 in
fvwm 1.xx and I had no problem with raising window being mistaken as a
move. Now, I have ClickTime set to 150 and fvwm (2.0.41) occasionally
(not always) starts a move even if I had released the mouse. Have you
tried adjusting the ClickTime parameter? If you had it set to something
like 750 then I'm sure you wouldn't have this problem, but then you get
a slight delay in your complex functions which could be annoying.

> Another one is: if you have set up
> EdgeScroll 100 100
> EdgeResistance 750 50
> OpaqueMoveSize 40
> and you move a nice solid xterm rapidly from the botton (for exemple)
> to the top of screen, stooping mouse on the very top, you can get in a
> situation where your xterm is blocked halfway, and then "jump" to the
> mouse position. But the most surprising thing is that if you release
[...]

This also happens to me, but it doesn't have to be a large opaque move,
moving a small rubber-band window will cause the mouse to freeze up and
jump, seems like an interaction with edge scroll where fvwm has to think
about whether to warp to the next page or not, causing the mouse to
freeze up.

Some other problems I'm currently having with 2.0.41:

1) clicking on a window in the pager will no longer raise the
corresponding window on the screen.

2) when a window is iconified, the icon label is truncated to the width
of the icon, and when you drag the icon the label is not drawn (this may
be a feature).

3) if you lower a window using the Lower command, then you cannot
immediately raise it by clicking inside the window.

4) if the CapsLock key is on then you can not click inside a window to
raise it.

> Well, fvwm is I think the better WM around the world, fast and nice
> looking. I hope this can help to spot the last bugs in it... keep with
> the fantastic work!

Agreed, fvwm has a lot of potential, but it still has some rough edges.
I hope we can smooth out those rough edges before a public release.
