>> From: "Mr. Graeme McCaffery" <mccaffer_at_eee.bham.ac.uk>
>>
>> I have mentioned this before, but my wish for fvwm is to have the
>> DEFAULT config file split into sections, to make it easier for
>> individual users to change things like style without having to copy the

KitS> In short I disagree. Apart from already having over 100 "dot"
KitS> files in my home directory I think that breaking up the
KitS> configuration file(s) is a bad idea.

I agree too. It's very easy for local installers to set up something
like this if they want, but I don't think it would be a good idea to
break it up by default.

KitS> I think it may be a better approach to allow the use of a common
KitS> system config file (eg /usr/local/lib/fvwm/fvwmrc). This file
KitS> would be read in as a precursor to any users' .fvwmrc file.

Fvwm will already do that now, basically. You could just tell users
to name their .fvwmrc files something like .fvwmrc-local, and make the
system one be .fvwmrc, which fvwm would read in and it could Read (or
FvwmM4 or FvwmCpp) .fvwmrc-local.

>> entire .fvwmrc file. This would make it easier for novice users to debug
>> faults in their configuration.

KitS> I dont think having more files and increasing the complexity
KitS> will help newbies. I think the biggest way to help them is to
KitS> improve the documentation.

I agree here wholeheartedly. That and supplying (or pointing to the
sites that have) all of those sample config files.
