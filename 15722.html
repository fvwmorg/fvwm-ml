>>>>> "jarl" == jarl <jarl_at_cpt.univ-mrs.fr> writes:

AL> But I think I'm getting the syntax wrong for telling fvwm to use another
AL> fvwmrc file.

jarl> Chuck:
>> fvwm -f "Read /apps/fvwm/system.fvwmrc"

jarl> Forgive me for rushing in on this, but isn't this very little
jarl> (ahem) elegant? I realize its value in having a consistent
jarl> syntax in which to run preprocessors, but IMHO it's not an
jarl> unreasonable request to be able to have fvwm understand the
jarl> intuitive

jarl> fvwm -f /a/path/with/.fvwmrc_in_it

jarl> and I see no problem in making fvwm understand that it is a
jarl> simple filename when it fails to be a fvwm command. The '-f', to
jarl> most unix users, _implies_ that its argument is a file, so if
jarl> you want to give an fvwm command it should rather be something
jarl> like fvwm -cmd "an fvwm command line" (Which is indeed
jarl> supported).

jarl> I guess there is legitimate reason for not letting fvwm
jarl> understand a simple filename, so would someone please enlighten
jarl> me?
....
jarl> Forgive me if there is some point in this I've completely failed
jarl> to grasp, it just seems so silly not to support "fvwm -f
jarl> filename".

Rob changed that in 2.xx for some reason long before I took over, so I
can't answer "why" with any certainty other than what you mentioned
above (consistent syntax to run the preprocessor modules). 1.xx used
to behave the way you would like. I've been thinking about changing
that back and it would be easy to do so.

I believe I put the -cmd parameter in there one day when toying with
the idea of changing that behavior of -f back to what you suggest (the
1.xx behavior) but I never finished.

So, for the sake of being/remaining consistent with other window
managers, I may do this for the next release. Does anyone have any
serious objections to this? I know it may mess people up for a few
moments until they read the ChangeLog, but I see no real problems
here.

Hey, it's still beta code so things like this are bound to happen. ;^)
