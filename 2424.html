"Adye, TJ (Tim) " <T.J.Adye_at_rl.ac.uk> writes:
> Hi,
>
> For some time, I have been using fvwm 2.3.6 built under Cygwin on Windows
> NT/2K with the Exceed 6.2 X-server (multi-window mode). I recently upgraded
> to the latest version (2.3.31). First let me congratulate you on a perfect
> build: it compiled and ran out of the box with no errors, the new sample-95
> configuration procedure ran like a dream, and a number of problems I had
> become accustomed to with the old version are now cured. Unfortunately a few
> new ones (not in fvwm 2.3.6) seem to have crept in:-
>
> 1) I use ClickToFocus. This mostly works correctly with X and native windows
> together, but if the active X-window is behind a native window, it is raised
> if the cursor moves over it (rather like MouseFocus). I tried turning on
> BugOpts RaiseOverNativeWindows, but this doesn't seem to make any difference
> (I guess it's switched on automatically). As one might expect, turning it
> off prevents X-windows being raised over the native window at all (even by a
> click).

Does clicking on the titlebar vs. clicking in the window act
differently? Have you looked at the various "ClickRaises" options?

> 2) Deiconising or switching to a window (by menu or taskbar) does an
> annoying full screen refresh. This doesn't happen when the window is
> selected by just clicking on it.

I don't see this. Are you using some kind of function to deiconify that
includes a refresh?

> 3) In my FvwmButtons (edited a little to conform to the new syntax) I have
> buttons like
>
> *BarButtons(Title XTerm, Icon rterm.xpm, Action `Exec "XTerm" exec
> xterm`)
>
> which I think should mean that the button remains shown as depressed until
> the XTerm window appears (this is what used to happen). Now the button is
> only shown depressed as long as mouse button is down. Is this a new bug, or
> am I using the wrong syntax?

I'm not sure, but it looks like there too many execs after action.
