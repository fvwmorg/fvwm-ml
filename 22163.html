| This could be a good idear, but I don't see the advantages over a new
| "good and clean" configuration language.

 Here is where we differ, because I feel that fvwm2's configuration
language is already highly optimized for its intended purpose and quite
good at being very expressive in a compact way. The only thing I think
is lacking is sophisticated functions, and one can get those without
throwing out the baby with the bathwater (and they need more than a
configuration language change anyways).

 If there are things that cannot currently be done in fvwm2's
configuration language, that's one thing. But to make the syntax more
complex and verbose and picky for no gain except it now being in some
general programming language seems to me to be a very perverse and
backwards thing to do.
