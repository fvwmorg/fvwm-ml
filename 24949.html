On Sun, 24 Oct 1999, Dan Espen wrote:

> "Jon A. Christopher" <jon_at_amanda.tamu.edu> writes:
> > Hello,
> >
> > I'm wondering why SaveUnders was removed from FVWM. It's there in 1.24r,
> > but not in the current version. This is a step backwards, IMHO, because
> > SaveUnders can sometimes be quite useful, particularly in applications
> > which have 3D graphics windows which are expensive to redraw. Because of
> > the lack of SaveUnders in FVWM, there are two expensive redraw operations
> > (~1/sec) apeice in my application everytime a dialog is popped up.
> > neither AfterStep, nor twm suffers from this problem, but FVWM is my WM of
> > choice.
> >
> > Please add the SaveUnders code back into FVWM. It's a simple code change,
> > and well worth having the ability to do so.
> >
> > Configuration: RH Linux 5.2, fvwm 2.0.46, XF86 3.3.5.
> >
> > P.S. Please CC any replies to me, since I'm not on this list.



> I assume when you mention the "current version", you don't mean
> 2.0.46. The current non-beta is 2.2.3 and the current beta is 2.3.10.

Nither the 2.0.46 version or the current CVS snapshot has SaveUnders.

> I'm curious, how do you know SaveUnders is removed?

Because I checked the source. SaveUnders is not allowed as an option in
the .fvwmrc files any more, nor is it set in add_window.c.

> Somewhere around 2.3.5 SaveUnders was removed as a configuration option,
> but I still see code that sets the SaveUnders attribute.

The only place in the current CVS version of FVWM where CWSaveUnder is
used is in builtins.c (turned off) and menus.c (turned on for fvwm menus).

If SaveUnder is good enough to use for FVWM menus, why can't I use it in
my applications?! Why was the SaveUnder code removed from add_window.c?
Since the WM reparents my windows *without* SaveUnders, nothing I can do
will work around this. Argh.
