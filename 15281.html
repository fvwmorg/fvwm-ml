John> 2. when i call up a new application now, i have to 'sit' it
John> down on the desktop myself. the apps used to place themselves

Again, RTFM. The FAQ included with the distribution (latest is
2.0.40) has fvwm 1.xx to 2.xx conversion chart. These particular ones
are controlled via the 'Style' command:

Style "*" RandomPlacement, SmartPlacement

John> fvwm2 has to have something good about it, but i don't see it
John> yet. any help would be appreciated!!

Fvwm2 does have a lot of good things about it, with many more coming
in the future. But the changes between the versions may make
conversion difficult.

Plus, since it's still beta code the docs and examples are not
completely up to date (so RTFM might not even help sometimes), and the
conversion script doesn't work 100%, which I meant to put a warning
about in the docs at some point. I'll make sure that this warning is
in the docs and printed by the conversion script as well for the next
release.

I'm trying to clean up the docs and build process so I can officially
release it so I can start doing some of the bigger projects on the
TO-DO list.

>>>>> "Frank" == Frank Weis <fweis_at_ulb.ac.be> writes:

Frank> the conversion script sets the modulepath to some strange
Frank> location. Set it right and fvwm2 will find your modules. Look
Frank> for the keyword FVWM_MODULEDIR in your dotrc.

You might want to start with a whole new .fvwm2rc file and add your
old customizations in the new format by hand.

If you want to see if it's finding FvwmAuto, try running FvwmTalk and
typing "Module FvwmAuto 250" in it and see if it gives an error
message. If it doesn't, then the module is being invoked and there is
another problem.
