I could use some of that myself...

>> For the next beta release I plan to make it look explicitly for
>> system.fvwm2rc in the 'FvwmDir' directory before it tries to read
>> on by default (haven't decided yet).

Lars> I think it may be a mistake to force a global config file to be
Lars> read. Or at least it will mean that the system admin can't put
Lars> too much of an environment there. For instance, he can't start
Lars> modules there, because not everyone will want them. A user
Lars> should be able to completely avoid a system.fvwmrc, IMHO.

Oh, definitely. I tried to word it to indicate that, but didn't
succeed. It'll either be added as an option to turn it on or off
(depending on what I make the default). Not sure which one I should
make the default though. I'm leaning towards default off.

>> Well, another way is you could have all of your users read a file at
>> the top of thier .fvwm2rc files that has all of your explicit menu
>> definitions and such:
>>
>> # top of ~/.fvwm2rc
>> Read /path/to/your/system.fvwm2rc
>> # rest of user's rc file follows, which may override these things...

Lars> I think this solution is not so hot. If the location or name of
Lars> the global file changes (and it _will_ change), every user
Lars> .fvwm2rc will have to be modified.

True. This was merely an interim solution suggestion.

Lars> My system.fvwmrc hack is better, IMHO, because it reads in files
Lars> from the user's directory.

A system.fvwm2rc that reads files from the user's directory would be
better, yes, but since fvwm doesn't currently look for a
system.fvwm2rc file, that's a minor problem... The -f flag is a
workaround to that though (or should be, but Carlos said that didn't
work for him for some reason). That was the reason I gave the not too
pretty solution as another option.
