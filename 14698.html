bruin> The linker complains that it cannot resolve the function
bruin> strdup() which is used in envvar.c in the libs directory.

That's right. Current FVWM (and modules) source assumes the existence
of following function related to string operation:

        strdup(), strncmp(), strstr(), strtol(), putenv()

My using-GNU-configure approach solves this problem, by adding
supplemental source to libs/ directory (which library sources come
from FreeBSD-2.0.5-RELEASE). Well, here is a patch of my poor work. If
you interested, check it and give me back some comments. I made this
patch (attached below) against to vanilla fvwm-2.0.38, then gzipped
and uuencoded.

This patch contains:

        + files to make 'configure' script.
                configure.in
        + files to make 'configure.h.in'.
                configure.h.{top,bot}
        + 'configure' script which is made by autoconf
        + 'configure.h.in' file which is made by autoheader
        + additional files for libfvwm
                libs/{strdup,strstr,strtol,putenv}.c
        + rename libs/strncmp.c to libs/mystrcasecmp.c and
          additional libs/strncmp.c which supports strncmp()
        + diffs for files which is already exists.
                fvwm/misc.h, libs/Imakefile,
                libs/{hostname,mygetostype,sleep}.c
                modules/FvwmBacker/FvwmBacker.c
                modules/FvwmConfig/WinBase.cc
                modules/FvwmDebug/FvwmDebug.c
                modules/FvwmWinList/FvwmWinList.c

Once you do a 'patch' to fvwm-2.0.38,

        + If you ONLY wanna do configure and how it works, first
          remove following bogus files.
                configure.h.{top,bot} configure.in
        + do 'chmod +x configure'
        + execute 'configure', and see what's done.
          configure script makes 'configure.h' file.
        + do edit configure.h as usual
        + do xmkmf or so, as usual

If you cannot run configure script, copy configure.h.in simply to
configure.h, and modify GNU autoconf stuff.
